# 需求完成度审查 (Requirement Audit)

## 概述

本 Skill 用于根据需求文档，对指定项目路径下的游戏项目进行需求完成度核查、潜在 Bug 风险分析和游戏内表现推演。

核心目标：基于项目中的真实证据（代码、蓝图、配置、资源、数据表、调用链），判断每个需求点是否已形成完整功能闭环。不做泛泛总结，不做简单关键词匹配即下结论。

## 核心原则（三大铁律）

1. **没有证据证明完成 → 不能判定为已完成**
2. **没有明确反证证明没做 → 不能判定为未完成**
3. **证据不足时 → 优先判定为疑似已完成或无法验证**

禁止行为：
- 主观臆断（必须有文件路径/函数/资源/代码行作为证据）
- 关键词命中即下结论（必须追踪完整调用链和功能闭环）
- 没搜到即判未完成（必须用同义词、英文、蓝图路径、配置名等多策略复搜）

## 完整工作流

`
需求文档 -> 拆解原子需求点 -> 生成搜索关键词
  -> 扫描项目代码和资源 -> 找候选实现
  -> 追踪调用链/资源引用/配置读取 -> 形成证据链
  -> 判断完成度 -> 检查潜在 Bug -> 输出 QA 可用报告
`

### 第 1 步：拆解原子需求点

将需求文档拆解为可验证的最小单元。每个原子需求点必须满足：
- 单一职责：一个点只描述一件事
- 可验证性：能通过代码/配置/资源确认是否实现
- 有明确边界：能区分做了和没做

格式：[R{n}] {需求点描述} —— {验收标准}

### 第 2 步：生成多策略搜索关键词

对每个原子需求点，生成不少于 4 组搜索关键词：

| 策略 | 说明 | 示例 |
|------|------|------|
| 中文关键词 | 需求中的中文术语 | 年成事件, 天灾 |
| 英文翻译 | 代码中常见英文命名 | HarvestEvent, Disaster |
| 同义词/别名 | 可能替代命名 | 丰收, 气候, CropEvent |
| 蓝图/配置路径 | 资源文件命名模式 | BP_Harvest, DT_Event |
| 配置表/枚举 | 配表名、枚举值 | EventType.Harvest |

### 第 3 步：分层次搜索

按以下层次依次搜索，确保覆盖完整：

1. **代码层**：.cpp/.h/.cs/.py/.lua — 逻辑实现
2. **蓝图/资源层**：.uasset/.umap — 蓝图节点、关卡
3. **配置层**：.json/.csv/.xml/.ini — 数据驱动配置
4. **配表层**：DDT / DataTable / CSV / JSON / Excel 导出表（如有）— 根据当前环境可用工具或文件搜索方式读取
5. **文档层**：.md/.txt — 设计文档、changelog

### 第 4 步：追踪证据链

搜到候选实现后，必须追踪完整证据链来判断是否形成功能闭环：

- **调用链**：谁调用了这个函数/类？它又调用了谁？
- **资源引用**：蓝图/代码引用了哪些资源？哪些资源引用了这个蓝图？
- **配置读取**：代码在哪里读取配置？配置表是否完整？
- **入口点**：功能是如何被触发的？（gamemode、controller、UI、事件等）

蓝图限制说明：如果只能看到 .uasset 文件路径，无法读取蓝图节点内部逻辑，则只能将其列为候选实现，不得直接判定为已完成。需要结合代码引用、接口调用或运行时证据才能确认。

### 第 5 步：判定完成度

基于证据链，按以下标准判定（详见 
references/completion_rules.md）：

| 判定 | 标准 | 证据要求 |
|------|------|----------|
| 已完成 | 功能完整闭环 | 与该需求适配的完整证据链 |
| 疑似已完成 | 有强证据但不完整 | 找到核心实现但缺部分入口/配置 |
| 部分完成 | 部分实现 | 找到部分功能但缺少关键环节 |
| 未完成 | 有明确反证 | 注释/空实现/TODO/明显缺失 |
| 无法验证 | 证据不足 | 多策略搜索均无可靠发现 |

判定约束：
- 判定为已完成：必须具备与该需求适配的完整证据链。证据链可以包括代码、蓝图、配置、数据表、资源引用、调用链、UI 绑定、关卡引用、运行日志或测试结果。
- 不是所有需求都必须同时具备代码、配置、调用链三类证据。如果某类证据对当前需求不适用，需要明确说明不适用的原因。
- 判定为未完成：必须有 TODO、注释、空类、空函数、配置缺失、入口未绑定、调用链断裂、正式流程未接入等明确反证。
- 证据不足时，优先判定为疑似已完成或无法验证，不要直接判定为未完成。

### 第 6 步：Bug 风险分析

对已完成、疑似已完成、部分完成的需求点，进一步分析潜在 Bug。

检查清单（详见 
references/bug_risk_rules.md）：
- 边界条件：空值/null 处理、数组越界、极端数值
- 状态一致性：初始化/重置/清理是否正确
- 并发/时序：多次触发的叠加、事件顺序
- 配置缺失：默认值是否存在、配置表行是否完整
- 引用完整性：跨表引用是否存在断链
- 数据持久化：存档/读档时数据是否正确保留
- UI/表现层：UI 刷新、输入阻断、资源加载、多分辨率

### 第 7 步：输出审查报告

严格按照 
references/report_template.md 中的模板格式输出。

## 搜索策略

1. 文本搜索：优先使用 
rg 搜索 .cpp .h .cs .py .lua .ini .json .csv .xml .md .txt 等文本文件中的关键词。
2. 文件名搜索：搜索蓝图、Widget、DataAsset、DataTable、Level、Map、Config 等资源文件名。
3. 配表搜索：如果当前环境提供 DDT、MCP、表格查询或 DataTable 读取工具，则使用工具读取表内容；如果没有对应工具，则退回到项目文件搜索和资源路径分析。
4. 调用链追踪：搜到类名、函数名、事件名、接口名后，需要继续反向搜索调用方和绑定方。
5. 资源引用追踪：搜到资源路径后，需要检查 ConstructorHelpers::FObjectFinder、LoadObject、TSoftObjectPtr、TSoftClassPtr、FSoftObjectPath、UAssetManager、CreateWidget、SpawnActor、DataTable RowHandle、GameplayTag、关卡引用和蓝图引用。
6. 蓝图限制说明：如果只能看到 .uasset 文件路径，无法读取蓝图节点内部逻辑，则只能将其列为候选实现，不得直接判定为已完成。

漏搜防御规则：
- 每个需求点用中文搜一次、英文搜一次、缩写搜一次
- 搜到类名/函数名后，以此为关键词再搜一次看谁在调用
- 搜到配置表名后，如工具可用则读取表内容确认
- 搜到资源路径后，检查代码中多种资源引用模式

## 注意事项

- 审查报告中的每一条结论都必须有证据支撑（文件路径 + 行号 或 配表名 + 行ID）
- 不要对未知模块做假设性判断——标记为无法验证而非强行判定
- 如果需求本身有歧义（未明确验收标准），在报告中标注需求不明确
- 项目路径下的 AGENTS.md 中的负责人信息可用于定位模块负责人

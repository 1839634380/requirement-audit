# 需求完成度审查 (Requirement Audit)

## 概述

本Skill用于根据需求文档对指定项目路径下的游戏项目进行需求完成度核查、潜在Bug风险分析和游戏内表现推演。

核心目标：基于项目中的真实证据（代码、蓝图、配置、资源、数据表、调用链），判断每个需求点是否已形成完整功能闭环。不做泛泛总结，不做简单关键词匹配即下结论。

## 核心原则（三大铁律）

1. **没有证据证明完成 → 不能判定为已完成**
2. **没有明确反证证明没做 → 不能判定为未完成**
3. **证据不足时 → 优先判定为疑似已完成或无法验证**

禁止行为：
- 主观臆断（必须有文件路径/函数/资源/代码行作为证据）
- 关键词命中即下结论（必须追踪完整调用链和功能闭环）
- 没搜到即判未完成（必须用同义词、英文、蓝图路径、配置名等多策略复搜）

## 完整工作流

`
需求文档 -> 拆解原子需求点 -> 生成搜索关键词
  -> 扫描项目代码和资源 -> 找候选实现
  -> 追踪调用链/资源引用/配置读取 -> 形成证据链
  -> 判断完成度 -> 检查潜在Bug -> 输出QA可用报告
`

### 第1步：拆解原子需求点

将需求文档拆解为可验证的最小单元。每个原子需求点必须满足：
- 单一职责：一个点只描述一件事
- 可验证性：能通过代码/配置/资源确认是否实现
- 有明确边界：能区分做了和没做

格式：[R{n}] {需求点描述} —— {验收标准}

### 第2步：生成多策略搜索关键词

对每个原子需求点生成不少于4组搜索关键词：

| 策略 | 说明 | 示例 |
|------|------|------|
| 中文关键词 | 需求中的中文术语 | 年成事件, 天灾 |
| 英文翻译 | 代码中常见英文命名 | HarvestEvent, Disaster |
| 同义词/别名 | 可能替代命名 | 丰收, 气候, CropEvent |
| 蓝图/配置路径 | 资源文件命名模式 | BP_Harvest, DT_Event |
| 配置表/枚举 | 配表名、枚举值 | EventType.Harvest |

### 第3步：分层次搜索

按以下层次依次搜索：

1. **代码层**：.cpp/.h/.cs/.py/.lua — 逻辑实现
2. **蓝图/资源层**：.uasset/.umap — 蓝图节点、关卡
3. **配置层**：.json/.csv/.xml/.ini — 数据驱动配置
4. **配表层**：DDT配表（如有）— search_sheet_by_name / search_string
5. **文档层**：.md/.txt — 设计文档、changelog

### 第4步：追踪证据链

搜到候选实现后，必须追踪完整证据链判断是否形成功能闭环：

- **调用链**：谁调用了这个函数/类？它又调用了谁？
- **资源引用**：蓝图/代码引用了哪些资源？哪些资源引用了这个蓝图？
- **配置读取**：代码在哪里读取配置？配置表是否完整？
- **入口点**：功能是如何被触发的？（gamemode、controller、UI、事件等）

### 第5步：判定完成度

基于证据链按以下标准判定：

| 判定 | 标准 | 证据要求 |
|------|------|----------|
| 已完成 | 功能完整闭环 | 代码+配置+资源+调用链完整 |
| 疑似已完成 | 有强证据但不完整 | 找到核心实现但缺部分入口/配置 |
| 部分完成 | 部分实现 | 找到部分功能但缺少关键环节 |
| 未完成 | 有明确反证 | 注释/空实现/TODO/明显缺失 |
| 无法验证 | 证据不足 | 多策略搜索均无可靠发现 |

判定约束：
- 判定为已完成：必须同时有代码、配置、调用链三类证据
- 判定为未完成：必须有TODO/注释/空类/空函数/配置缺失等明确反证
- 证据不足时优先判定为疑似已完成或无法验证

### 第6步：Bug风险分析

对已完成、疑似已完成、部分完成的需求点进一步分析潜在Bug。

检查清单（详见 references/bug_risk_rules.md）：
- 边界条件：空值/null处理、数组越界、极端数值
- 状态一致性：初始化/重置/清理是否正确
- 并发/时序：多次触发的叠加、事件顺序
- 配置缺失：默认值是否存在、配置表行是否完整
- 引用完整性：跨表引用是否存在断链
- 数据持久化：存档/读档时数据是否正确保留
- UI/表现层：UI刷新、输入阻断、资源加载、多分辨率

### 第7步：输出审查报告

严格按照 eferences/report_template.md 中的模板格式输出。

## 搜索策略

1. **文本搜索**：g 搜索 .cpp/.h/.cs/.py/.ini/.json 中的关键词
2. **DDT配表搜索**：mcp__ls4ddt__search_string 和 mcp__ls4ddt__search_sheet_by_name
3. **文件名搜索**：Get-ChildItem -Recurse -Filter "*Keyword*"
4. **调用链追踪**：对找到的函数名再次搜索追踪上下游

漏搜防御规则：
- 每个需求点用中文搜一次、英文搜一次、缩写搜一次
- 搜到类名/函数名后以此为关键词再搜一次看谁在调用
- 搜到配置表名后用 get_sheet_data 读取表内容确认
- 搜到资源路径后检查 .cs/.cpp 中是否有 LoadObject 等引用

## 注意事项

- 审查报告中的每一条结论都必须有证据支撑（文件路径+行号 或 配表名+行ID）
- 不要对未知模块做假设性判断——标记为无法验证而非强行判定
- 如果需求本身有歧义在报告中标注需求不明确
- 项目路径下的AGENTS.md中的负责人信息可用于定位模块负责人

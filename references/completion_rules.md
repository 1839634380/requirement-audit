# 完成度判定规则 (Completion Judgement Rules)

## 判定矩阵

| 证据维度 | 已完成 | 疑似已完成 | 部分完成 | 未完成 | 无法验证 |
|---------|--------|-----------|---------|--------|---------|
| 代码实现 | 完整 | 核心逻辑有 | 部分有 | 无或仅有 TODO | 不确定 |
| 配置/配表 | 完整 | 部分有 | 部分有 | 缺失 | 不确定 |
| 资源/蓝图 | 完整 | 不确定 | 部分有 | 缺失 | 不确定 |
| 调用链/入口 | 连通 | 不确定 | 部分连通 | 未连通 | 不确定 |

## 每类判定的必要条件

### 判定为已完成

必须具备与该需求适配的完整证据链。证据链可以包括：
- 代码实现（函数/类/模块）
- 蓝图节点/事件图
- 配置表或数据表
- 资源文件（.uasset/.umap）
- 调用链（谁触发、谁被调用）
- UI 绑定（Widget/UMG）
- 关卡引用（Level/Map 中放置的 Actor）
- 运行日志或测试结果

**不是所有需求都必须同时具备代码、配置、调用链三类证据。** 如果某类证据对当前需求不适用，需要明确说明不适用的原因。

### 判定为疑似已完成

满足以下之一：
1. 找到核心实现代码，但配置或入口不完整
2. 找到完整配置和调用链，但核心代码只找到部分
3. 找到与需求高度相关的代码和资源，但无法 100% 确认对应关系
4. 只能看到 .uasset 文件路径，无法读取蓝图节点内部逻辑

**重要**：判定为疑似已完成的，必须在证据链中说明缺什么、为什么缺。

### 判定为部分完成

满足以下之一：
1. 找到部分功能点实现，但明确缺少另一个子功能
2. 核心框架存在，但具体逻辑是空实现
3. 代码中有条件编译或功能开关未启用

### 判定为未完成

必须满足以下之一：
1. 搜索到明确的 TODO/FIXME 注释标注功能未实现
2. 函数体为空/return 空值/仅打印日志
3. 配置表中对应行完全不存在
4. 需求对应的模块/文件夹完全不存在
5. git log 中明确记录暂未实现或未提相关提交
6. 入口未绑定、调用链断裂、正式流程未接入

### 判定为无法验证

满足以下之一：
1. 需求描述过于模糊，无法确定搜索关键词
2. 项目文件类型不支持搜索（纯二进制、加密包等）
3. 需求涉及外部服务/第三方 SDK，本地无法验证
4. 多策略搜索后仍无有效命中，且不能排除存在可能
5. 蓝图 .uasset 文件无法解析内部逻辑，且没有对应的 .cpp/.h 实现

## 防误判规则

| 误判类型 | 原因 | 防御规则 |
|---------|------|---------|
| 没做报完成 | 看到相似代码/命名就下结论 | 必须追踪完整证据链 |
| 做了报没做 | 没搜到关键词就判定未完成 | 必须用中/英/缩写/同义词复搜；必须搜配表和资源路径 |
| 部分当全部 | 找到一个函数就认为全做了 | 必须对照需求的每个子功能点逐一验证 |
| 全当部分 | 识别不出已实现的功能闭环 | 如果证据链完整，即使命名不一致也应判为已完成 |
| 蓝图当完成 | 看到 .uasset 路径就判定已完成 | 蓝图文件只能列为候选，必须结合代码引用或运行时证据 |

## 证据权重

| 证据类型 | 权重 | 说明 |
|---------|------|------|
| 完整函数体实现 | ★★★ | 最强证据 |
| 配置表中的完整行 | ★★★ | 与代码配套则极强 |
| 蓝图节点/事件图内容 | ★★★ | 需要能读取内部逻辑 |
| 蓝图/.uasset 文件路径 | ★☆☆ | 仅路径不足以证明实现 |
| 资源文件存在 | ★★☆ | 需结合代码引用 |
| 函数声明/接口定义 | ★☆☆ | 仅声明不足以证明实现 |
| 注释/TODO 提及 | ★☆☆ | 可作为反证 |
| 相似命名 | ☆☆☆ | 仅命名相似不是证据 |

# 完成度判定规则 (Completion Judgement Rules)

## 判定矩阵

| 证据维度   | 已完成     | 疑似已完成  | 部分完成     | 未完成         | 无法验证 |
| ------ | ------- | ------ | -------- | ----------- | ---- |
| 代码实现   | 完整      | 核心逻辑存在 | 部分存在     | 无实现或仅有 TODO | 不确定  |
| 配置/配表  | 与需求适配   | 部分存在   | 部分存在     | 明确缺失        | 不确定  |
| 资源/蓝图  | 与需求适配   | 有候选资源  | 部分存在     | 明确缺失        | 不确定  |
| 调用链/入口 | 连通      | 不完全确定  | 部分连通     | 未连通         | 不确定  |
| 游戏内表现  | 可解释预期表现 | 基本可推测  | 只能解释部分表现 | 无法产生预期表现    | 不确定  |

## 每类判定的必要条件

### 1. 判定为已完成

必须具备与该需求适配的完整证据链。证据链可以包括：

- 代码实现，例如函数、类、模块、组件、Subsystem。
- 蓝图节点或事件图。
- 配置表、DataTable、DataAsset、CSV、JSON。
- 资源文件，例如 `.uasset`、`.umap`。
- 调用链，例如谁触发、谁被调用、调用后发生什么。
- UI 绑定，例如 Widget、UMG、按钮事件、数据刷新。
- 关卡引用，例如 Level / Map 中放置的 Actor。
- 运行日志或测试结果。

不是所有需求都必须同时具备代码、配置、调用链三类证据。
如果某类证据对当前需求不适用，需要明确说明不适用的原因。

判定为已完成时，必须能说明该需求的完整闭环：

```
触发入口 -> 条件判断 -> 核心逻辑 -> 数据变化 -> 表现反馈 -> 状态保存或同步 -> 异常处理或重复触发保护
```

如果某些环节不适用于当前需求，需要说明原因。

### 2. 判定为疑似已完成

满足以下任一情况，可以判定为疑似已完成：

1. 找到核心实现代码，但配置、入口、调用链或运行时证据不完整。
2. 找到完整配置和调用链，但核心逻辑只能看到部分实现。
3. 找到与需求高度相关的代码和资源，但无法完全确认对应关系。
4. 只能看到 `.uasset` 文件路径，无法读取蓝图节点内部逻辑。
5. 找到通用系统实现，逻辑上可能覆盖需求，但还不能确认该需求流程实际接入。
6. 找到 UI 或配置入口，但不知道是否有对应的后端逻辑。
7. 找到资源文件，但不知道是否被代码或蓝图实际引用。
8. 通过命名、路径、注释推测大概率是实现，但缺少直接证据。

判定为疑似已完成的，必须在证据链中明确说明：
- 已经找到什么证据。
- 缺什么证据。
- 为什么缺这些证据。
- 为什么不能判定为已完成。

### 3. 判定为部分完成

满足以下任一情况，可以判定为部分完成：

1. 找到需求中部分功能点的完整实现，但明确缺少另一个子功能。
2. 核心框架存在，但部分具体逻辑是空实现。
3. 代码中有功能开关、条件编译或配置开关未启用。
4. 只实现了客户端或服务器单侧逻辑，另一侧缺失。
5. 只实现了正向逻辑，没有处理错误、边界、空值、重复触发、取消、恢复等分支。
6. 只实现了数据存储，没有实现数据读取和使用。
7. 只实现了单个场景，没有处理跨场景、读档、重新开始等持续性状态。
8. 只做了单人逻辑，没有处理多人同步。
9. 只做了正常流程，没有处理取消、失败、中断、死亡、切图等异常路径。
10. 只做了资源或配置，但没有确认被正式流程读取。

### 4. 判定为未完成

只有在找到明确反证时，才能判定为未完成。

明确反证包括：

1. 搜索到明确的 TODO / FIXME / Not Implemented 注释，说明功能未实现。
2. 函数体为空、直接 return 空值、仅打印日志、没有实际逻辑。
3. 配置表中需求明确要求的行、字段或 key 完全不存在。
4. 需求对应的模块、文件夹、资源、入口在项目中明确不存在。
5. git log、注释、任务记录或变更说明中明确说明该功能暂未实现、未接入、被废弃或被禁用。
6. 入口未绑定、调用链断裂、正式流程未接入。
7. 找到相关实现，但实现内容与需求要求明显相反或明显不满足需求。
8. 只存在测试代码、Demo 代码或废弃代码，正式流程没有使用。

注意：

- 不能因为"没有搜到关键词"就判定为未完成。
- 不能因为"C++ 中没有实现"就判定为未完成。
- 不能因为"git log 未提到相关提交"就判定为未完成。
- 不能因为"蓝图无法读取内部逻辑"就判定为未完成。
- 如果没有明确反证，应判定为"疑似已完成"或"无法验证"。

### 5. 判定为无法验证

满足以下任一情况，可以判定为无法验证：

1. 需求描述过于模糊，无法确定验收标准或搜索关键词。
2. 项目文件类型不支持搜索，例如纯二进制、加密包。
3. 需求涉及外部服务、服务端代码、第三方 SDK，本地项目无法验证。
4. 多策略搜索后仍无有效命中，且不能排除存在其他实现路径。
5. 蓝图 `.uasset` 文件无法解析内部逻辑，且没有对应的代码引用、运行日志或测试结果。
6. 数据表、DataAsset、配置内容不可见。
7. 运行环境缺失，无法确认功能是否在运行时接入。
8. 调用链或资源引用链中断，无法继续确认。

## 防误判规则

| 误判类型  | 原因                    | 防御规则                               |
| ----- | --------------------- | ---------------------------------- |
| 没做报完成 | 看到相似代码或相似命名就下结论       | 必须追踪完整证据链                          |
| 做了报没做 | 没搜到关键词就判定未完成          | 必须用中文、英文、缩写、同义词、配置名、资源名复搜          |
| 部分当全部 | 找到一个函数就认为全做了          | 必须对照需求的每个子功能点逐一验证                  |
| 全部当部分 | 识别不出已实现的功能闭环          | 如果证据链完整，即使命名不一致，也应判定为已完成           |
| 蓝图当完成 | 看到 `.uasset` 路径就判定已完成 | 蓝图文件只能列为候选，必须结合代码引用、蓝图导出、运行日志或测试结果 |
| 配置当完成 | 看到配置表行就判定已完成          | 必须确认配置是否被正式流程读取和使用                 |
| 接口当完成 | 看到接口声明就判定已完成          | 必须确认接口实现和调用方                       |

## 证据权重

| 证据类型                | 权重  | 说明               |
| ------------------- | --- | ---------------- |
| 完整函数体实现             | 高   | 强证据              |
| 配置表中的完整行            | 高   | 与代码读取逻辑配套时是强证据   |
| 蓝图节点 / 事件图内容        | 高   | 需要能读取内部逻辑        |
| 运行日志 / 自动化测试结果      | 高   | 能证明运行时行为         |
| 调用链完整连通             | 高   | 能证明功能被正式流程触发     |
| 蓝图 / `.uasset` 文件路径 | 低   | 仅路径不足以证明实现       |
| 资源文件存在              | 中   | 需要结合代码或蓝图引用      |
| 函数声明 / 接口定义         | 低   | 仅声明不足以证明实现       |
| 注释 / TODO 提及        | 低   | 可作为反证线索，但需要结合上下文 |
| 相似命名                | 无   | 仅命名相似不是证据        |
